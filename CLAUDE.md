## Architecture

Go TUI for managing git worktrees across repositories. Uses **Bubble Tea** (MVC pattern) with **lipgloss** for styling.

**Data flow:** `main` reads `WORKTREE_ROOT` env var → `scanner.Scan()` discovers repos → `model.New(repos)` creates Bubble Tea model → `Init()` fires async `fetchBranches` → `ui.Render()` draws two-pane layout.

- **`cmd/wtui/main.go`** — Entry point. Wires scanner output into the Bubble Tea program (alt-screen mode).
- **`actions/`** — Executes git operations. `RemoveWorktree`/`ForceRemoveWorktree` run `git worktree remove` (with/without `--force`). `DeleteBranch`/`ForceDeleteBranch` run `git branch -d`/`-D`. `DropStash` runs `git stash drop stash@{N}`. `OpenTerminal(path)` runs `open -a Terminal <path>`. `OpenVSCode(path)` runs `code <path>`.
- **`scanner/`** — Discovers git repos under `WORKTREE_ROOT` (default `~/dev`), up to 2 levels deep. Excludes `*-worktrees` dirs. Detects `.git` directories only (`.git` files are worktree markers and are excluded so worktree checkouts don't appear as separate repos). Returns repos sorted case-insensitively.
- **`gitquery/`** — Queries git data. `ListBranches(repoPath)` uses `git for-each-ref` to discover all local branches, then per branch queries upstream status, ahead/behind counts, unpushed commits, and worktree dirty status (files changed, lines added/deleted). Branches can carry multiple worktree paths. `BranchDiff(worktreePath)` returns `git diff HEAD` for a worktree. `ListStashes(repoPath)` runs `git stash list --format=%gd%x00%ai%x00%s`. `StashDiff(repoPath, index)` runs `git stash show -p stash@{N}`. Only list-level failures are hard errors; per-item failures silently default to zero values.
- **`model/`** — Bubble Tea Model with two-pane focus system. `activePane` (0=left repos, 1=right content) determines which pane receives key input. `tab` toggles focus between panes. **Destructive mode** (`destructive` bool, default false): `Shift+D` toggles; when off, `d` key is a no-op (read-only safety gate). `Shift+D` is blocked while any overlay is active. **Left pane** (`activePane==0`): `up/down` (`j/k`) navigate repos (resets right-pane cursors, fires fetch); `repoScroll` tracks viewport offset via `ensureRepoVisible()` using `height - ui.RepoContentOverhead`; all other keys are no-ops except `tab`, `Shift+D`, and `q/esc`. **Right pane** (`activePane==1`): full key handling — `up/down` navigate branches/stashes, `left/right` (`h/l`) and `1/2` switch modes, `enter` opens diffs (gated to dirty+worktree branches or stashes), `d` deletes with confirmation (requires destructive mode), `t`/`c` open terminal/VSCode at worktree path. Branch cursor (`branchSelected`) is an absolute index into flattened rows (all branches selectable). `branchScroll` tracks viewport offset via `ensureBranchVisible()` using `height - ui.BranchContentOverhead` (status bar + 2 borders + mode header with separator). `stashScroll` tracks stash viewport offset via `ensureStashVisible()` using `ui.StashLineCount()` to account for long messages that wrap to two lines. On delete failure, `DeleteFailedMsg` triggers a red-text force confirm dialog. `d` in stash mode → `DropStash` confirm. Result messages update state with stale-result protection via `isCurrentRepo()`. Confirm overlay intercepts all keys: `y`/`enter` executes, `n`/`q`/`esc` cancels. Code is split into focused handler methods: `handleConfirmKey`, `handleLeftPaneKey`, `handleOverlayKey`, `handleRightPaneKey`, plus action-specific methods (`handleEnter`, `handleDelete`, `confirmBranchDelete`, `confirmStashDrop`). Tests split across `model_test.go` (navigation), `model_action_test.go` (actions/confirm), `model_view_test.go` (view output).
- **`ui/`** — Stateless rendering. Two-pane layout with lipgloss borders: left pane (30 chars, full 4-sided border) + right pane (full 4-sided border). Active pane border = blue (color "12"), inactive = grey (color "238"). In destructive mode, right pane border = red (color "9") regardless of focus. `RenderParams` includes `ActivePane int`, `Destructive bool`, `RepoScroll int`, `BranchScroll int`, and `StashScroll int`. Left pane scrolls via `RepoScroll`; long repo names are truncated with `truncateToWidth` to prevent line wrapping. Right pane has a mode header as its first content line (`renderModeHeader` shows `[1] branches  2 stashes`). Below the header: mode 1 shows branch details with viewport scroll via `BranchScroll`, mode 2 shows stash list with viewport scroll via `StashScroll`. Stash entries with long messages always wrap to two lines (first line: date + message start, second line: indented continuation); `StashLineCount(msg, paneWidth)` returns 1 or 2 based on message width vs `StashPrefixWidth` (15). `splitAtWidth` splits messages at the visible-width boundary. Branch pane highlighting uses absolute branch index (all branches selectable). Full-screen diff overlay replaces two-pane when active. Confirmation dialog overlay (`Overlay==3`) shows a centered prompt; `ConfirmForce=true` renders it in red for force-delete dialogs. `RenderStatusBar(width, mode, overlay, activePane int, destructive bool)` renders hints only (no mode tabs — those moved to the right pane header). `tab: pane` and `q/esc: quit` appear first in all modes. In read-only mode (destructive=false), `D: destructive mode` hint appears; delete/drop hints are hidden. In destructive mode, `d: delete`/`d: drop` hints render in red. Mode 1 shows indicator legend; action hints (`t: terminal`, `c: code`) only appear when right pane is active. Branch status indicators: green `✔` (clean/even), yellow `●` (ahead/behind with `+N/-N` counts), red `●` (dirty with `N files +X/-Y`), purple `●` (no upstream). Indicators stack. Worktree branches show a location label **after** the status indicators: `[root]` if checked out at the repo root, `[<path>]` for additional worktrees. Non-worktree branches show no label. Up to 5 unpushed commits shown with overflow. Constants: `LeftPaneWidth = 30`, `RepoContentOverhead = 3`, `BranchContentOverhead = 5`, `StashPrefixWidth = 15`.

## CI

CI runs on push to `main` and PRs targeting `main`. Checks: `gofmt`, `make test`, `make build`.

## Testing

Tests use real temp directories with actual `.git` dirs/files — no mocks. Scanner tests create nested repo structures; model tests simulate key messages via Bubble Tea's `Update()`. Gitquery tests create real git repos with remotes, commits, and worktrees to verify dirty/clean, ahead/behind, and unpushed detection.
